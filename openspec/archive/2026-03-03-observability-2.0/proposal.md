# 提案：企业级可观测性 (Observability 2.0)

## 1. 背景与动机
- **背景**: 当前 Polyshift 框架仅通过简单的 `log.Printf` 输出日志，缺乏结构化信息，难以进行自动化分析和告警。且由于微内核架构涉及多语言插件和跨进程通信，缺乏全链路追踪能力导致故障定位困难。
- **问题**:
  1. **日志非结构化**: 难以搜索特定请求或错误。
  2. **缺乏全链路追踪**: 无法追踪请求在 Core 与 Plugin 之间的流转路径及耗时。
  3. **监控指标缺失**: 缺乏关键的运行时指标（QPS、延迟、错误率、资源使用）。
- **目标**: 构建“可观测性 2.0”体系，实现日志 (Logs)、追踪 (Traces)、指标 (Metrics) 的深度关联，提升故障排查效率和系统透明度。

## 2. 范围与约束
- **包含范围**:
  - **结构化日志**: 采用 Go 标准库 `log/slog` (Go 1.21+)，统一输出 JSON 格式日志。
  - **分布式追踪**: 集成 OpenTelemetry Go SDK，实现 Core -> gRPC -> Plugin 的全链路追踪 (W3C Trace Context)。
  - **指标监控**: 通过 OpenTelemetry Prometheus Exporter 暴露 `/metrics` 接口。
  - **上下文透传**: 在 HTTP Headers 和 gRPC Metadata 中透传 TraceID。
  - **关联性**: 确保每条日志都携带 TraceID 和 SpanID，方便关联查询。
  - **配置管理**: 在 `config.yaml` 中新增 `observability` 配置段，支持动态调整采样率 (Sampling Rate) 和日志级别。
- **不包含范围**:
  - 搭建集中式日志收集系统 (ELK/Loki) 和监控面板 (Grafana) —— 本次仅负责生成和暴露数据。
  - 对所有存量插件的强制改造（优先改造 Core 和示例插件）。

## 3. 关键指标 (成功标准)
- **日志关联**: 100% 的请求日志携带 TraceID。
- **追踪覆盖**: 覆盖 HTTP 入口、内部路由、gRPC 调用、插件处理全流程。
- **性能开销**: 引入可观测性组件后，P99 延迟增加不超过 5ms。

## 4. 风险与缓解措施
- **风险**: 引入 OpenTelemetry SDK 可能导致二进制体积显著增加。
  - *缓解*: 按需引用必要的 OTel 库，或考虑使用精简版实现。
- **风险**: 高并发下大量的 Trace 采样可能影响性能。
  - *缓解*: 支持配置采样率 (Sampling Rate)。
