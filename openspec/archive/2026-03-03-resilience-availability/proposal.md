# 提案 (Proposal): 弹性与高可用性增强 (Resilience & Availability)

## 1. 背景与问题 (Context & Problem)
当前的微内核服务框架虽然实现了多语言插件的基础加载与通信，但在生产环境下的稳定性（Stability）和容错能力（Fault Tolerance）方面存在显著短板。
*   **缺乏自愈能力**: 当插件进程因 bug 崩溃、内存泄漏或死锁时，Core 无法自动检测并恢复，导致服务长时间不可用。
*   **缺乏流量保护**: 当插件响应过慢或频繁报错时，Core 缺乏熔断机制，可能导致请求堆积，最终拖垮整个网关。
*   **路由性能瓶颈**: 随着插件和路由规则数量的增加，现有的基于 Gin 静态路由或简单遍历的动态路由匹配机制性能会线性下降，且不支持高效的运行时更新。

*   **用户故事 (User Story)**:
    *   作为**运维人员**，我希望系统能够自动重启崩溃的插件，以便在深夜故障时无需人工干预即可恢复服务。
    *   作为**平台架构师**，我希望在某个插件故障时能将其隔离，不影响其他健康插件的运行。
    *   作为**核心开发者**，我希望路由匹配性能不受插件数量影响，保持恒定的低延迟。

## 2. 目标 (Goals)
*   [ ] **自动保活 (Watchdog)**: 实现插件进程的健康监控与自动重启机制（支持指数退避策略）。
*   [ ] **熔断降级 (Circuit Breaker)**: 在 Core 侧实现针对插件的熔断器，当错误率或延迟超标时自动切断流量。
*   [ ] **高性能动态路由**: 替换现有的路由实现，采用 Radix Tree (基数树) 算法实现 O(1) 复杂度的动态路由匹配，并支持并发安全的原子更新。

## 3. 非目标 (Non-Goals)
*   [ ] **业务级监控**: 本次迭代不包含具体的业务指标（如订单量、交易额）监控。
*   [ ] **分布式限流**: 仅关注单机维度的保护，不涉及分布式集群的限流同步。

## 4. 核心价值 (Value Proposition)
*   **高可用性 (High Availability)**: 通过自愈机制，将服务的平均恢复时间 (MTTR) 从分钟级（人工介入）降低到秒级（自动恢复）。
*   **稳定性 (Stability)**: 通过熔断机制防止故障扩散，确保核心网关在部分插件故障时仍能存活。
*   **可扩展性 (Scalability)**: 高性能路由确保系统能支撑数千个插件和路由规则，而不会引入显著延迟。

## 5. 风险与缓解 (Risks & Mitigation)
*   **风险 1**: **重启风暴 (Restart Storm)**。如果插件因配置错误或外部依赖故障而启动即崩溃，无限重启会消耗大量系统资源。
    *   **缓解**: 引入**指数退避 (Exponential Backoff)** 算法（如 1s, 2s, 4s... max 60s），并设置最大重启次数阈值。
*   **风险 2**: **路由更新的并发竞争**。在请求处理过程中更新路由表可能导致数据竞争或读取到不一致的状态。
    *   **缓解**: 使用 Copy-On-Write (COW) 策略或原子指针交换 (`atomic.Value`) 来更新路由树。

## 6. 成功指标 (Success Metrics)
*   [ ] **自愈时间**: 插件崩溃后，Core 能在 5秒内 检测到并完成重启。
*   [ ] **路由延迟**: 在 1000+ 路由规则下，路由匹配耗时 < 0.1ms (P99)。
*   [ ] **故障隔离**: 单个插件 100% 错误率时，Core 的 CPU/内存占用无显著飙升，其他插件请求成功率 100%。
